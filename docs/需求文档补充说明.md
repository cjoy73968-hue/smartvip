# 需求文档补充说明

> 本文档补充《微信小程序会员管理系统需求方案》，覆盖业务逻辑确认、遗漏功能补充、技术架构建议。

---

## 一、账户体系（新增）

### 1.1 账户结构

会员账户分为**两个独立子账户**，分别管理：

| 账户类型 | 来源 | 用途 | 可提现 | 可退款 |
|---------|------|------|--------|--------|
| 充值余额 | 用户主动充值 | 消费、提现 | ✓ | ✓ |
| 返利余额 | 推荐返利 | 仅限消费 | ✗ | ✗ |

### 1.2 消费扣款优先级

消费时**优先扣除充值余额**，不足时自动扣除返利余额。

```
示例：
用户充值余额：200元
用户返利余额：50元
用户消费120元

扣款顺序：
1. 扣除充值余额 120元
2. 充值余额剩余：80元
3. 返利余额保持：50元（未动用）
```

### 1.3 账户余额展示

用户端个人中心分开展示两个余额，清晰告知余额来源和可用范围。

## 二、推荐裂变机制（补充）

### 2.1 推荐关系绑定规则

- 一个用户只能被一个推荐人推荐
- 用户使用谁的推荐码，即确认与该推荐人绑定
- 绑定后关系永久有效，不可变更

### 2.2 返利比例

- 由商家在后台设置（范围由系统配置，建议上限15%）
- 用户可在小程序查看当前返利比例

### 2.3 返利到账时机

被推荐会员消费并确认收货后7天，返利自动进入推荐人账户。

### 2.4 返利可见性

推荐人查看返利明细时，**不显示被推荐人信息**，仅显示返利金额和到账时间。

### 2.5 返利追回机制

若被推荐人发生退款，且退款时返利已到账，系统自动从推荐人账户追回已发放返利。

```
示例：
用户B消费100元 → 推荐人A获得5元返利（7天后到账）
用户B申请退款，商家同意退款100元
→ 若5元已到A账户，系统自动从A账户扣除5元
→ 若5元尚未到账，则不发放
```

## 三、订单与履约（补充）

### 3.1 消费订单流程

```
下单支付 → 商家发货 → 7天无退货自动确认 → 商家到账 → 7天后返利到推荐人账户
```

**关键节点：**
- 商家发货后，订单状态变为"配送中"，由商家送货成功后修改状态"配送完成"
- 7天内无退货申请，系统自动确认收货
- 商家实收金额 = 订单金额 - 返利金额
- 返利在确认收货7天后到账

### 3.2 退款规则（统一）

- **消费订单**：一律不退款，若产品质量问题，线下协商换货
- **充值余额**：可提现至微信账户，货物退款只可退款至余额
- **返利余额**：不可退款、不可提现，仅可用于消费

### 3.3 退货与返利追回

发生退款时：
1. 退款金额原路退回（充值余额退回充值余额，返利余额退回返利余额）
2. 已发放返利无论是否到账，均需追回
3. 若用户返利余额不足，系统记录欠款，后续消费时优先扣除

## 四、礼品派送模块（重大补充）

### 4.1 新增：用户地址管理

用户入驻后需完善收货地址，用于礼品派送和消费配送。

**地址字段：**
- 收货人姓名
- 联系电话
- 省/市/区/详细地址
- 是否设为默认地址

**功能要求：**
- 支持添加多个收货地址（家/公司等）
- 支持修改/删除地址
- 支持设置默认地址

### 4.2 配送范围限制

- 仅支持**同城配送**
- 商家在后台配置可配送城市
- 用户输入地址时，系统判断是否在配送范围内
- 若为偏远乡镇/农村，提示"该地址需自提，配送服务暂不可用"
- 用户可选择其他地址或选择继续下单（转为自提）

### 4.3 配送费用

同城免费配送，不收取额外配送费。

### 4.4 礼品派送模式

礼品派送支持**两种独立模式**，用户可自行选择：

#### 模式一：周期自动派送（默认）

- 用户从礼品库选择一个礼品作为"周期派送礼品"
- 系统按商家配置的派送周期（每日/每周/每月等）自动触发
- 费用：仅支付差价（用户自选礼品价格 - 系统默认礼品价格）
- 若差价≤0，用户无需付费

```
示例：
用户选择礼品A（价格150元），系统默认礼品B（价格100元）
每次周期派送，用户仅需支付差价50元
```

#### 模式二：一次性下单派送

- 用户可随时选择任意礼品进行一次性下单派送
- 费用：**全额支付礼品价格**，不与默认礼品比较差价
- 与周期派送**完全独立**，分开结算

```
示例：
用户选择礼品C（价格200元）一次性派送
用户支付全额200元，不涉及默认礼品
```

#### 核心区别

| 维度 | 周期自动派送 | 一次性下单派送 |
|------|------------|--------------|
| 触发方式 | 系统按周期自动触发 | 用户随时下单 |
| 费用计算 | 差价模式 | 全额支付 |
| 与默认礼品关系 | 比较差价 | 无关 |
| 订单类型 | 周期派送订单 | 一次性派送订单 |

### 4.5 价格变动处理

商家修改礼品价格后：
- **已生成订单的周期**：按原价格执行
- **下一个周期**：按新价格计算差价

**价格变动通知：**
- 用户自选礼品价格发生变动时，微信通知用户："您选择的礼品[名称]价格已调整，请留意"

### 4.6 派送周期说明

- 派送周期由商家在后台配置（每日/每周/每月等）
- 当前周期进行中时，商家修改派送周期**不影响进行中的周期**
- 修改后的周期从下一个周期生效

### 4.7 礼品派送取消规则

#### 周期自动派送

- 差价扣除后（订单生成）：不可取消，不可退款
- 质量问题处理：线下换货，不走退款流程

#### 一次性下单派送

- 付款后不可取消，不可退款
- 质量问题处理：线下换货，不走退款流程

### 4.8 礼品派送与消费订单的区别

| 场景 | 费用模式 | 确认收货 | 返利时机 |
|------|---------|---------|---------|
| 消费订单 | 全额 | 7天无退货自动确认 | 确认后7天 |
| 礼品派送（周期） | 差价 | 无需用户确认，商家确认送达后7天 | 送达后7天 |
| 礼品派送（一次性） | 全额 | 无需用户确认，商家确认送达后7天 | 送达后7天 |

## 五、商家后台功能补充

### 5.1 新增：配送操作功能

**派送订单管理：**
- 查看所有待处理派送订单（区分周期派送订单/一次性派送订单）
- 点击"确认送达"完成履约
- 订单列表显示：收货人、地址、电话、礼品、**金额**（周期显示差价，一次性显示全额）

### 5.2 新增：库存管理功能

**礼品库存：**
- 显示每个礼品当前库存
- 库存预警：低于设置阈值时提醒商家
- 库存不足时处理规则：切换默认礼品 / 暂停派送 / 商家手动补货

### 5.3 新增：配送配置功能

**配送设置：**
- 配送城市：选择可配送的城市
- 超范围处理：不允许下单 / 允许下单需自提

## 六、非功能需求补充

### 6.1 系统稳定性要求

- 充值、扣费、返利、派送、消息推送核心功能零卡顿
- 数据实时同步，避免漏单、错账
- 定时任务需加锁，防止重复执行

### 6.2 账户安全要求

- 余额操作需事务支持，保证数据一致
- 返利计算需幂等，防止重复发放
- 退款时需检查返利余额是否充足，不足时记录欠款

### 6.3 消息通知要求

微信模板消息通知场景：
- 新派送订单通知商家
- 用户余额不足提醒
- 返利到账提醒
- 会员到期提醒
- 礼品价格变动提醒
- 营销推送消息

## 八、技术架构建议

### 8.1 技术选型

| 层级 | 技术 | 说明 |
|------|------|------|
| 用户端 | 微信小程序（原生） | 微信授权/支付/地图 |
| 商家端 | Vue3 + TypeScript + Element Plus | PC浏览器访问 |
| 后端 | Java 17 + Spring Boot 3 | 成熟稳定，复杂业务友好 |
| ORM | MyBatis-Plus | 轻量，SQL可控 |
| 数据库 | MySQL 8.0 | 事务支持好 |
| 缓存 | Redis 7 | session/定时锁/热数据缓存 |
| 文件存储 | 七牛云OSS | 礼品图片存储 |
| 部署 | 腾讯云轻量应用服务器（2核4G） + Docker | 性价比高 |

### 8.2 系统模块划分

```
用户端模块（小程序）
├── 登录/授权
├── 会员入驻/地址管理
├── 充值/消费/余额展示
├── 推荐裂变/海报分享
├── 礼品选择
├── 订单查看
└── 消息通知

商家端模块（PC后台）
├── 会员管理
├── 订单管理（含派送确认）
├── 礼品管理（含库存）
├── 数据报表
├── 营销推送
└── 系统配置（含配送城市）

定时任务（后端核心）
├── 会员有效期检查（每日）
├── 礼品派送触发（按配置时间）
├── 返利到账处理（每周）
└── 数据报表生成（每月）

微信集成模块
├── 支付（充值/退款）
├── 模板消息
├── 登录授权
└── 小程序跳转
```

### 8.3 预计工期

- 用户端（小程序）：3-4周
- 商家端（PC后台）：4-5周
- 后端服务：4-5周
- **总计：约12-14周（3-3.5个月）**
