## Context

微信小程序会员管理系统是一套完整的私域会员运营解决方案，包含用户端（微信小程序）和商家端（PC后台）。系统核心功能包括会员入驻裂变、双账户余额管理、周期礼品派送、推荐返利等，采用微服务架构设计，支持高并发和事务一致性。

系统设计约束：
- 单商家定制：系统仅为一家商户服务，充值直接进入商家微信商户号
- 线下实体场景：同城免费配送，支持自提
- 会员订阅制：需购买会员才可享受周期礼品派送和推荐返利权益

技术选型：
- 用户端：微信小程序（原生），支持双入口（小程序码/公众号跳转）
- 商家端：Vue3 + TypeScript + Element Plus，PC浏览器访问
- 后端：Java 17 + Spring Boot 3 + MyBatis-Plus
- 数据库：MySQL 8.0（事务支持）
- 缓存：Redis 7（Session、定时锁、热数据缓存）
- 文件存储：七牛云OSS（商品图片）
- 部署：腾讯云轻量应用服务器 + Docker

## Goals / Non-Goals

**Goals:**
- 实现完整的会员入驻流程（微信授权、手机绑定、地址管理、推荐关系绑定）
- 实现双账户体系（充值余额/返利余额）及其消费优先级规则
- 实现会员订阅机制（购买/续费/有效期/自动订阅/到期提醒）
- 实现推荐裂变返利（推荐码、上下级绑定、7天到账、返利追回）
- 实现周期礼品派送（差价模式）和一次性派送（全额模式）
- 实现商家PC后台（会员/订单/商品/配置/报表/营销推送）
- 实现微信服务通知（模板消息推送）
- 保证余额操作的事务一致性和返利计算的幂等性

**Non-Goals:**
- 不支持多商家（单商家定制）
- 不支持跨城配送（仅限同城）
- 不支持会员间直接转账
- 不支持微信外支付
- 不实现完整的CRM系统（仅基础营销推送）

## Decisions

### 1. 双入口架构（小程序 + 公众号）

**决策**：用户端同时支持小程序码扫码入驻和公众号菜单跳转两种方式。

**理由**：
- 小程序码适合线下场景（地推、传单、门店海报）
- 公众号适合已有粉丝的私域流量（图文链接、菜单栏）
- 两者可覆盖所有微信私域流量入口

**实现**：
- 小程序：生成带参数的二维码，扫描后携带场景值跳转
- 公众号：菜单配置小程序跳转路径，图文消息嵌入小程序卡片

### 2. 数据库设计

**决策**：采用MySQL 8.0，使用InnoDB引擎，所有表支持事务操作。

**核心表结构**：

- 统一表命名规范，便于维护

### 3. 双账户余额设计

**决策**：用户余额分为两个独立账户（recharge_balance/rebate_balance），消费时优先扣充值余额。

**扣款逻辑**：
```
消费金额 = A
充值余额 = R
返利余额 = B

if R >= A:
    recharge_balance -= A
else:
    recharge_balance = 0
    rebate_balance -= (A - R)
```

**理由**：
- 充值余额可提现、可退款，返利余额仅限消费不可提现
- 分账管理便于商家结算和财务对账
- 优先级规则确保用户优先消耗可提现余额

### 4. 返利到账机制

**决策**：被推荐人消费后，商家确认发货，7天无退货自动确认收货后才将返利发放给推荐人。

**时序**：
```
T+0: 被推荐人消费，订单创建，返利状态=待到账
T+0: 商家发货
T+7: 系统检查无退货，确认收货，返利状态=已到账
T+7: 推荐人账户增加返利余额
```

**补偿机制**：
- 若7天内发生退款，且返利已到账，系统自动追回
- 若返利余额不足，记录欠款，后续消费优先扣除

**理由**：
- 7天等待期覆盖主流退货周期，降低坏账风险
- 延时到账便于处理售后问题，避免已返利后又退货的困境
- 追回机制确保商家不会因退货而损失已发放的返利

### 5. 周期礼品派送逻辑

**决策**：系统按商家配置的周期自动触发派送，用户选择礼品与默认礼品比较差价。

**差价计算**：
```
price_diff = user_selected_gift_price - default_gift_price

if price_diff <= 0:
    用户无需付费，系统派送默认礼品（若用户余额不足也派送默认礼品）
else:
    if 用户余额 >= price_diff:
        扣除差价，派送用户选择礼品
    else:
        派送默认礼品，发送微信通知用户余额不足
```

**礼品变更规则**：
- 本周期未派送：变更后立即生效
- 本周期已派送：下一周期生效

**理由**：
- 差价模式降低用户实际支出，提高会员续费率
- 默认礼品兜底确保派送不中断
- 余额不足时自动切换默认礼品，保证用户体验和商家利益

### 6. 定时任务设计

**决策**：使用Redis分布式锁确保定时任务不重复执行。

**核心定时任务**：
```
每日任务：
├── 会员有效期检查（检查到期会员，停止权益）
├── 自动订阅扣款（检查即将到期且开启自动订阅的会员）
├── 会员到期提醒（提前30天发送微信通知）
├── 返利到账处理（检查确认收货7天的订单，发放返利）

按配置时间触发：
└── 礼品派送触发（按商家配置的派送周期和时间）
```

**锁机制**：
```
SET lock_key unique_value NX PX 30000
# 获取锁后执行任务，执行完毕删除锁
# 若服务器宕机，30秒后锁自动释放，其他服务器可抢锁恢复
```

**理由**：
- 分布式锁防止多实例部署时任务重复执行
- 宕机恢复机制确保定时任务不漏单
- 按商家配置的时间触发，适应不同时区需求

### 7. 微信消息通知

**决策**：使用微信模板消息，按场景订阅。

**通知场景**：
| 场景 | 接收方 | 触发时机 |
|------|--------|---------|
| 新派送订单 | 商家 | 周期派送触发/用户下单 |
| 余额不足提醒 | 用户 | 差价扣除失败 |
| 返利到账提醒 | 用户 | 确认收货7天后 |
| 会员到期提醒 | 用户 | 到期前30天 |
| 礼品价格变动 | 用户 | 用户选择的礼品价格调整 |

**理由**：
- 模板消息需要用户授权（每次下单时订阅）
- 营销推送需用户明确同意接收
- 不同场景触发不同模板，确保信息相关性

### 8. 部署架构

**决策**：腾讯云轻量应用服务器（2核4G）× 2 + Docker Compose。

**架构**：
```

                           │
    ┌──────────────────────┼──────────────────────┐
    │                      │                      │
┌───┴───┐            ┌─────┴─────┐          ┌────┴────┐
│ 小程序 │            │  API Server │         │ Web后台 │
│ 用户端 │            │  (Java)     │         │ (Vue)   │
└───────┘            └─────┬─────┘          └────┬────┘
                           │
                    ┌──────┴──────┐
                    │             │
               ┌────┴────┐  ┌────┴────┐
               │  MySQL  │  │  Redis  │
               └─────────┘  └─────────┘
```

**理由**：
- 轻量应用服务器性价比高，适合中小商家
- Docker Compose简化部署，易于迁移
- Nginx统一入口，便于后续扩展HTTPS

## Risks / Trade-offs

**[风险] 微信支付回调失败**
→ **缓解**：微信支付采用异步回调，失败时记录日志并重试（最多3次），人工对账兜底

**[风险] 定时任务重复执行**
→ **缓解**：Redis分布式锁 + 任务执行记录检查，双重保险

**[风险] 用户余额数据不一致**
→ **缓解**：余额操作使用事务 + 乐观锁（version字段），失败时回滚重试

**[风险] 商品库存超卖**
→ **缓解**：数据库层面加锁（SELECT FOR UPDATE），库存扣减原子操作

**[风险] 返利计算错误（重复发放）**
→ **缓解**：返利计算幂等设计（基于订单号去重），已发放的跳过

**[权衡] 实时性 vs 可靠性**
→ 返利7天到账是可靠性优先的设计，确保退货纠纷可处理

**[权衡] 单商家 vs 多商家**
→ 当前仅支持单商家，降低系统复杂度，利于快速交付

## Open Questions

1. **微信支付分账**：是否需要实现分账功能（将部分收入分给推荐人）？
   - 当前方案：返利从商家收入中扣除（系统内记账），不涉及真实分账
   - 若需要微信侧分账，需申请分账API

2. **退款流程**：消费订单和礼品派送均不支持退款，若出现质量纠纷如何处理？
   - 当前方案：线下协商换货，不走退款流程
   - 是否需要预留退款接口以应对特殊状况？

3. **自动订阅失败重试**：自动订阅扣款失败后，是否需要重试？
   - 当前方案：余额不足则自动订阅失败，发送通知用户手动续费
   - 若用户一直不续费，是否需要持续提醒？

4. **商家结算周期**：商家的收入（充值金额 - 返利支出）多久结算一次？
   - 当前方案：充值直接进入商家微信商户号，返利从商家收入中扣除
   - 是否需要提供商家对账单功能？
